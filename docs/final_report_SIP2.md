## 2. プロジェクト概要
**説明**: どんな人が利用するのか、その人たちが解決したい課題や得られる価値を、引用つきでデータに基づいて記載する。

**記載例**:
- 対象ユーザ：
  地域の高齢者福祉施設で働くスタッフ（介護士・看護師・事務員）、1施設あたり平均20〜30名規模。PCおよびタブレット端末を日常業務で利用[^1]。
- 解決したい課題：
  入所者のケア記録が紙媒体・口頭で共有されるため、情報の見落としや重複作業が発生する。施設間で状況を共有する必要があるが、手間と時間がかかる[^2][^3]。
- 得られる価値：
  統一フォーマットの電子記録で、データ検索性・履歴追跡が容易になる。また、複数施設間でリアルタイム共有可能なツールにより、転所・交流時の情報移行がスムーズになる。
- アプリ概要：
  クラウドベースのケア記録共有アプリ。タブレットPCで使える入力画面を備え、入所者の記録（体温・服薬・食事など）を日次で記録。情報は施設内外でアクセス可。過去履歴のグラフ表示機能あり。セキュリティ対応（SSL、ロールベースアクセス制御）を想定。

## 3. プロジェクトの成果物
### 3.1 UI と主要な機能
**説明**：  
作成した画面や主要機能を簡潔にまとめる。単なる図やスクリーンショットだけではなく、「どのような目的で作り」「どのようにユーザの課題を解決するのか」を文章で明確に説明すること。  
特に主要機能については、2章で述べた課題にどのように対応し、どんな価値を提供するのかを具体的に示すこと。

**記載例（UIは省略していますので注意してください）**：
- **UI 概要**  
  本システムは、タブレット操作を前提に大きめのボタンや高コントラスト配色を採用し、介護スタッフが片手でも操作しやすいUIを目指した。主要な画面は以下の3種類で構成される：  
  1. 記録入力画面（体温・食事・服薬などを簡易入力）  
  2. 利用者一覧画面（検索フィルタ付き）  
  3. 記録閲覧・グラフ画面（日付別の推移を視覚化）

- **主要機能と課題解決への寄与**
  - **(1) 記録入力機能**  
    現場で即時に記録を入力できる機能。紙で発生していた「記入漏れ」や「後からの思い出し記録」を防ぎ、データはクラウドに自動保存されるため、ほかのスタッフがリアルタイムで共有可能となる。
  - **(2) グラフ表示機能**  
    体温・食事量などを自動的にグラフ化し、利用者の状態の変化をひと目で把握できる。これにより異変の早期発見や申し送り時の情報共有が容易になる。
  - **(3) 利用者情報検索**  
    利用者名・日付・記録項目で絞り込み可能。紙のファイルを探す必要がなく、情報の見落とし防止と作業時間の短縮につながる。
  - **(4) 権限管理（管理者／一般）**  
    閲覧・編集の可否を役割ごとに分離することで、誤編集や情報漏洩のリスクを軽減する。

### 3.2 残った課題と解決案
**説明**：  
完成しなかった機能や、現時点で不十分な点について、理由とともに表形式で整理する。  
その上で、表の内容を基に文章で補足説明を行うこと。  
改善案については技術面・体制面の両方から具体的に示すこと。

**記載例**：
最終的な結果を表1に示す。まず一つ目の課題は...
以上より、現段階では基本的なCRUDとグラフ表示機能は実装できたものの、運用において重要な「即時性」「通知性」「安全性」にまだ課題が残っている。  
特に現場の利用環境ではモバイル回線が一般的であるため、通信最適化は急務である。
API のレスポンス削減やキャッシュ導入を行うことで、データ取得速度の改善が期待できる。また、アラート機能は利用者の危険サインの早期把握に不可欠なため、次期開発では優先度を上げて実装する必要がある。
権限管理についても、介護士・看護師・管理者で必要な情報が異なるため、ロール定義を再整理し、画面単位で柔軟に権限を設定できるよう改善していく予定である。

表1: 課題と解決案の整理
| 課題 | 現状の問題点 | 完成しなかった理由／不十分な理由 | 改善案（今後の方針） |
|------|--------------|----------------------------------|-------------------------|
| モバイル通信時の動作が遅い | 読み込み時間が長く、現場での即時入力が難しい | APIの最適化が未実施でデータ量が多い | レスポンス圧縮、不要データ削減、キャッシュ導入 |
| アラート機能が未実装 | 危険値や急変時に自動通知ができない | 優先度が低く後回し | 閾値設定UIの追加、定期チェック処理の実装 |
| 権限管理の細分化不足 | 一部画面で管理者と一般が混在 | 実装時間不足・仕様整理の遅れ | ロール定義の再整理、画面単位で権限付与 |

# 4. プロジェクトの進行

## 4.1 全体的な進行の感想
**説明**：  
プロジェクトの進行状況について、ガントチャートを用いた計画管理を踏まえて振り返る（ガントチャートも適宜貼ってください）。  
スケジュール通りに進んだ点だけでなく、遅延が発生した理由や、計画変更のプロセス、チーム内でのコミュニケーション方法（ミーティング頻度・共有ツールなど）についても記述すること。  
単に「大変だった」「楽しかった」ではなく、計画と実行の差分から学んだことを書き、次回のプロジェクトに活かす視点を含める。

**記載例**：  
今回の開発では、初期段階でガントチャートを作成し、「調査→要件定義→UI設計→実装→テスト」という流れで計画を立てた。  
前半の要件定義に予想以上の時間がかかったことで、UI設計の開始が約1週間遅れたが、週1回のミーティングで進捗を共有し、役割の再調整を行うことで後半の遅れを挽回できた。  
また、GitHub Projects を利用し、タスクを可視化したことで作業の抜け漏れが減り、ガントチャートと実績を比較しながら進めたことは、メンバー全員の見通しの共有に役立った。

## 4.2 担当者と貢献したこと
**説明**：  
各メンバーの担当領域と、実際にどのような貢献をしたかを記述する。  
作業分担だけでなく、チームへのサポート面（レビュー、調整役、資料作成など）も含めて良い。  
また、「担当した機能」だけを書くのではなく、ガントチャート上でどの工程を主に担当したか（要件定義・設計・実装・テスト等）を書くと良い。

**記載例**：  
- **マグロ（リーダー）**：  
  要件定義とスケジュール管理を中心に担当。ガントチャートの作成と更新を行い、週次ミーティングを主導して進捗の把握と調整を行った。  
  API連携部分のレビューも担当し、チーム全体の開発方針の統一に貢献した。

- **はまち（バックエンド）**：  
  データベース設計、記録APIの実装、テストデータ作成を担当。  
  ガントチャートの実装フェーズを主に担当し、遅延の発生時には優先度の高いタスクを切り分けて処理した。

- **ナマズ（フロントエンド）**：  
  記録入力画面・一覧画面の実装、レスポンシブ対応を担当。  
  UI設計が遅れた際、Figmaとの連携作業を並行して進めることで全体の遅延を最小限に抑えた。

- **タイ（UIデザイン）**：  
  FigmaによるUI設計とプロトタイプ作成を担当。  
  要件定義フェーズでのヒアリング内容を整理し、設計資料としてドキュメント化することで、実装メンバーが迷わずに作業できる環境を整えた。
